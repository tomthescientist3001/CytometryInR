01 - Installing R Packages

Background

Week 1: During this first session, we will ensure that everyone’s computer gets properly setup, as well as start building individual participants familiarity with the software infrastructure that they will be using throughout the rest of the course. Namely, you will need to install R, Rtools, Positron, Quarto, and Git; as well as setup and link to a GitHub account. This session also touches on concepts behind version control and how to install R package from the various repositories (CRAN, Bioconductor, GitHub).

Walk Through

R as a software has it’s origins in the 90s. It was created, as most good things, in New Zealand, derriving from an earlier programming language S. When we install R, we are installing a collection/module of interconnected packages that together make up base software, ie. base R. These were primarily focused on statistics, which is why often refer to R as a statistical programming language.

This is not the case today, and part of that derives from packages. An R package in essence is a bunch of code, that extends the functionality of base R, to be able to do something novel. And novel is an understatement. We have R packages for flow cytometry, we have R packages for econometrics, for ecology, you, for those nice graphs we see during election season, we even have color-palette packages so that your graphs can have Wes Anderson, Studio Ghibli, or Bluey colors.

This, and a vibrant community around it, have helped sustain R ecosystem for many years (refer to the various projects that are particularly useful). Within this context, where does Cytometry in R fit in? Well, most tie in to the Bioconductor project. Originating in 2004, originally as a way to try to be able to work with DNA microarrays, it now occupies a large collection of interoperable packages ranging from RNA-based to spatial to flow cytometry.

How do we get these R packages that do the various useful things? From repositories. The concept is partially borrowed from the Linux community, where community-driven projects like Debian set up similar repositories for downloading software. For a lot of R packages, the main repository is CRAN.

When a package is hosted on CRAN, the install occurs with the following set of commands in the console window:

install.packages("dplyr")

Within R code, what we see above is called a function. There is the name of the function (in this case install.packages), followed by the argument within the parenthesis, in this case dplyr, the name of the package, shown within quotation marks.

In context of a fresh R installation, the function will check first whether you already have the package on your computer, if not, it will fetch the package from the repository (assuming you are connected to the internet) and carry out the installation. In the context of CRAN and Bioconductor for Windows and Mac, it is installing pre-compiled packages in the form of binaries. However, it is also possible to build the package from the source code (in Linux until recently, this was the only option). This is typically handled in the background by the installer.

For the Bioconductor project, they have their own package to manage the installations, BiocManager. This is available via CRAN, and needs to be installed before you are able to use it to install packages that are stored in the Bioconductor repositories:

install.packages("BiocManager")

Once installed, we need to make the respective package available to our computer. If you check at the moment, you will see that within your environment, only base R is active:

#environment

Let’s change this by now using the library function on BiocManager

library(BiocManager)

On using the library call, the functions contained within the BiocManager package are made available to be used, we can see this if we check the loaded packages again

#environment

To learn more about what a package does, we have a couple options.

We can write the name of the package, and add two colons (::), which will result in the name of the individual functions present within the package to become visible. In positron, if we hover over these, we can see the help file information detailed.

Alternatively, we can do this by adding a single ? in front of the function name if we have already called that package via library, or ?? if we haven’t called via library.

?select
??select

For installation of packages available via the Bioconductor project, we will use BiocManager’s install function:

install("flowCore")

Once again, when installing a package we surround it within quotation marks. However, when loading it with the library call, we don’t use quotation marks

library(flowCore)

In the process, you will see that we saw various messages during the course of the installation. Most packages are not self-contained, but depend on other packages to carry out their task. These other packages are known as dependencies. Often times, if the installation of a package fails, it is because you are missing one of these other dependencies, which should be installed first. To do so, read the error message, and attempt to locate which package is contributing to what you are witnessing.

Finally, outside CRAN and Bioconductor, a bunch of R packages can also be found on GitHub. These are generally newer packages, whose developers are either still working on polishing up the package for submission to either CRAN or Bioconductor, or who have decided not to bother and just make it available separately.

To install packages from GitHub, you will need the remotes package, which is packaged along with the in the devtools package. The installation command in this case is:

install_github("DavidRach/Luciernaga")

Where install_github is the function name from remotes, and within the quotation marks before the / you have the GitHub Username, and after the / the name of the package.

When installing from GitHub, you are downloading the R package in it’s source code form (as can be viewed here). To be executable on your computer, this source code needs to be compiled into a binary (similar to how packages are transferred to your computer by CRAN and Bioconductor). For this, on Windows, you need to have Rtools installed, and the developer tools if you are on Mac.

When this is the case, what you need to do is specify a function within that package. To do so, add two colons after the package name.

You can also notice, that within Positron, you can see similar information if you hover over the name of the function.

Within the context of R, a function is a piece of code that carries out a particular task. We will cover these in more detail in a few weeks. Basically, given a set of arguments, they perform their code, and return a value.

For the following:

TotalCells <- Counts(Bcells=5000, Tcells=4000)

Where Counts is the (), and Bcells and Tcells are the provided arguments. The <- is an assignment arrow, this is used to save the output to a variable.

Once this code block is run, this variable populates the session. This is now available and can be used in combination with other arguments

TotalCells + 1000

One way of seeing what a particular set of code does, is to type out the variable immediately after the code is run

Alternatively, if you highlight the code on the right side of the assignment arrow, and run ctrl+enter, you can see the values appear, but not have them present yet in the session.

Alternatively, if you run a command and have an object that you don’t want present, you can remove it. One way of doing this is the rm function, providing it the name of the variable that you inadvertently created

You will notice, this has now left the session window. If you type it into the console, or Run the cell again, it will fail to locate and return an error.

Alternatively, you can delete the variable directly from the session window above.
